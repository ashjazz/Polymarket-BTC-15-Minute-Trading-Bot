# 研究文档：价值投资策略 + 分批止盈系统

**日期**: 2026-03-10
**状态**: 完成

## 1. 现有代码库分析

### 决策：保留 NautilusTrader 框架
**理由**：
- NautilusTrader 提供了成熟的交易框架，包括订单管理、风险引擎、数据引擎
- Polymarket 适配器已经实现并经过测试
- WebSocket 连接和行情订阅机制已经稳定运行

**替代方案考虑**：
- 直接使用 py_clob_client：需要重新实现订单管理、状态跟踪等基础功能，增加开发复杂度
- 使用其他交易框架（如 Backtrader）：不支持 Polymarket，需要自定义适配器

### 决策：移除多信号融合系统
**理由**：
- 新策略不依赖信号处理器（SpikeDetection, Sentiment 等）
- 价值投资策略的核心是价格触发，而非多因子信号
- 移除可简化代码，减少外部依赖

**替代方案考虑**：
- 保留信号处理器作为辅助：增加复杂度，与核心策略逻辑冲突
- 将信号处理器作为可选模块：当前不需要，可在未来扩展

### 决策：保留 Redis 模式切换
**理由**：
- 允许在运行时切换模拟/实盘模式
- 不需要重启服务即可切换模式
- 已有的基础设施，维护成本低

## 2. 策略实现模式研究

### 决策：状态机模式管理持仓生命周期
**理由**：
- 持仓有明确的状态转换（OPEN → CLOSED_TP1/CLOSED_TP2/CLOSED_TP3/CLOSED_SL/CLOSED_EOD）
- 时间阶梯止盈需要精确的检查点管理
- 状态机使代码更清晰，易于测试和调试

**状态转换图**：

                                      │ 价格进入入场区间
                                      ▼
                    ┌─────────────────────────────────────────┐
                    │                 OPEN                     │
                    │  (entry_price, entry_time=T0)           │
                    └─────────────────┬───────────────────────┘
                                      │
          ┌───────────────────────────┼───────────────────────────┐
          │                           │                           │
          ▼                           ▼                           ▼

### 决策：配置类使用 dataclass + 环境变量
**理由**：
- Python 3.11 原生支持 dataclass
- 环境变量配置符合 12-factor app 原则
- 类型注解提供 IDE 支持和运行时验证

**替代方案考虑**：
- 使用 Pydantic：增加依赖，当前简单配置不需要复杂验证
- 使用 YAML 配置文件：环境变量更简单，适合单一服务

### 决策：检查点使用定时器 + 事件驱动混合模式
**理由**：
- 止盈检查点是时间驱动的（T+2min, T+4min, T+6min）
- 止损是价格驱动的（实时监控）
- 混合模式兼顾两种需求

**实现方案**：
- 使用 `asyncio` 定时器进行止盈检查
- 在 `on_quote_tick` 中实时检查止损条件
- 避免阻塞主事件循环

## 3. 双向价格监控研究

### 决策：同时订阅 YES 和 NO 代币
**理由**：
- NautilusTrader 支持多工具订阅
- 两个代币的价格是互补的（YES + NO ≈ 1.0）
- 同时监控可以捕捉任一侧的价值机会

**实现方案**：
```python
# 在 on_start 中订阅两个代币
self.subscribe_quote_ticks(self._yes_instrument_id)
self.subscribe_quote_ticks(self._no_instrument_id)

# 在 on_quote_tick 中根据 instrument_id 判断是哪个代币
def on_quote_tick(self, tick):
    if tick.instrument_id == self._yes_instrument_id:
        self._check_entry("UP", tick.mid_price)
    elif tick.instrument_id == self._no_instrument_id:
        self._check_entry("DOWN", tick.mid_price)
```

### 决策：每市场限制一个持仓
**理由**：
- 简化风险管理
- 避免 UP 和 DOWN 同时持仓的对冲情况
- 符合策略设计（价值投资，不是高频交易）

## 4. 错误处理研究

### 决策：保留现有的断路器和重试机制
**理由**：
- `circuit_breaker.py` 已实现连接断路器
- `connection_config.py` 已有超时配置
- 这些机制对新的策略同样适用

### 决策：订单失败不阻塞策略
**理由**：
- 市场订单可能因流动性不足被拒绝
- 记录失败原因，允许下次检查点重试
- 不影响其他市场的交易

## 5. 测试策略研究

### 决策：单元测试 + 集成测试
**理由**：
- 单元测试覆盖策略核心逻辑（入场条件、止盈止损）
- 集成测试验证与 NautilusTrader 的交互
- 模拟模式作为端到端测试

**测试重点**：
1. 入场条件判断（价格区间、时间窗口）
2. 止盈检查点触发
3. 止损实时触发
4. 状态转换正确性
5. 配置加载和验证

## 6. 性能考虑

### 决策：轻量级状态管理
**理由**：
- 只跟踪当前市场的持仓状态
- 历史持仓持久化到 JSON 文件
- 不需要数据库，减少复杂度

### 决策：异步执行不阻塞行情处理
**理由**：
- 交易决策在 executor 中执行
- 行情处理保持轻量
- 避免错过价格更新
